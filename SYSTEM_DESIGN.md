# System Design Write-up

## Complaint history model

Every complaint carries a mutable "current state" (`status`, `priority`, `is_closed`) on the `complaints` row, plus an append-only `complaint_history` table that records one row per status transition: the resulting status, an optional note, the actor (id + name, captured at write time so history reads correctly even if a user is later renamed/removed), and a timestamp. The complaint is created with an initial `Open` history row ("Complaint raised") so the log is always complete from the start, and every subsequent admin status change adds another row rather than mutating a single field. This gives a full, ordered audit trail per complaint without needing event-sourcing machinery — a simple one-to-many relationship, fetched with the complaint and sorted by timestamp. Denormalizing `actor_name` onto each history row (rather than only storing `actor_id`) was a deliberate trade-off: it costs a little redundancy but keeps the history immutable and readable in perpetuity. Priority changes are *not* logged in this table since priority is an administrative triage attribute rather than part of the resident-facing lifecycle; only status changes are historicized, keeping the trail focused on "what happened to my complaint" rather than internal bookkeeping.

Resolution is a one-way gate: once `status` is set to `Resolved`, the API sets `is_closed = true` and `resolved_at`, and the update endpoint rejects further status changes on a closed complaint. This matches the spec's "once resolved, it is closed" rule and avoids needing a separate re-open workflow that wasn't requested.

## Overdue detection

Overdue is computed, not stored. Storing a boolean flag risks going stale (a complaint would silently stop being "overdue" only when something re-touches the row). Instead, `days_open` and `is_overdue` are derived on every read: `days_open` is `(now − created_at).days` for open complaints, or `(resolved_at − created_at).days` for resolved ones (so a complaint that took 10 days to resolve stays informative even after closing, rather than resetting to "0 days open"). A complaint is `is_overdue` only if it's still open *and* `days_open` meets or exceeds a configurable threshold. The threshold lives in a generic `settings` key-value table (currently one key, `overdue_threshold_days`) rather than an environment variable alone, specifically so the admin can tune it live from the dashboard without a redeploy — the `.env` value is only the seed default for a fresh database. In the admin complaint list, results are sorted with overdue complaints first, then by priority (High → Medium → Low), so the most urgent, longest-neglected issues surface without the admin having to filter manually. This sort happens in the API layer after computing overdue status, keeping the frontend a thin renderer of already-prioritized data.

## Photo handling

Photos are optional and attached via `multipart/form-data` on complaint creation. The backend validates extension (jpg/png/gif/webp) and enforces a size cap (default 5MB, configurable) before ever touching disk, rejecting bad uploads with a clear 400 rather than silently failing. Accepted files are renamed to a random UUID + original extension and written to a local `uploads/` directory, which FastAPI serves as static files at `/uploads/<filename>`; the complaint row stores only that relative URL, not the file itself, keeping the database lightweight. This local-disk approach is intentionally the simplest thing that works for the assignment's scope and for typical low-traffic society deployments, and it's isolated behind one function (`_save_photo`) — swapping it for S3/Cloudinary later is a localized change, not a rearchitecture, and the README flags this explicitly since most free hosting tiers have ephemeral disks.

## Notification flow

Email is decoupled from business logic through a small `send_email()` helper that calls the Resend HTTP API. Two flows trigger it: (1) whenever an admin changes a complaint's status, the resident is emailed with the new status and any note, sourced straight from the same request that writes the history row, so the email and the audit trail can never disagree; (2) whenever an admin posts a notice with `is_important = true`, every resident is emailed a short summary. Both are best-effort — if `RESEND_API_KEY` is unset or the API call fails, the app logs the email content instead of sending it and the request still succeeds, so email configuration issues never block core complaint/notice functionality. This also makes local development and testing frictionless, since no real email account is needed to exercise the full status-update and notice-posting flows.
